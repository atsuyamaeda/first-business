## 1. このプロジェクトは何か

東京都心10区（**江東・台東・港・目黒・品川・世田谷・杉並・新宿・中野・豊島**）の抹茶カフェ競合データベースを作成中。抹茶D2C/店舗ビジネスの Phase 3 戦略策定（出店エリア選定・ポジショニング）のインプットとなる。

**包含基準**: 
1. 抹茶メニュー3品以上の店、**または**
2. 抹茶の品種（銘柄・産地）が複数選べる店（品数少なくても専門店扱い）、**または**
3. スターバックス全店（10区内の全店舗を脅威的な競合として全件収集）

他チェーン（ドトール・タリーズ・猿田彦・コメダ等）は基準1・2のいずれかを満たさない限り無視。

**参考**: 10区のスタバ総数 = 約176店（港44/新宿34/世田谷23/豊島17/品川15/江東14/目黒10/台東10/杉並6/中野3）

## 4. 具体的な作業のやり方（復帰時のワークフロー）

### Step 0: コンテキスト復元
1. このファイル（STATUS.md）を読む
2. [schema.md](schema.md) を読む（列定義・包含基準・★「取得できた情報は全て格納する」ルールを把握）
3. [matcha_cafes.md](matcha_cafes.md) を読む（現状のデータ）
4. auto memory（`~/.claude/projects/-Users-maedaatsuya-projects-first-business/memory/MEMORY.md`）を確認
5. 「2. 次にやるべきこと」を実行

### Step 1: 店名候補収集
1. **エリア分割**: [schema.md](schema.md) のエリア定義に従って区を街区粒度で分割
2. **WebSearch 網羅検索**: 各エリア × キーワード配列 `["抹茶","matcha","和カフェ","茶寮","茶房","日本茶","甘味処"]` で検索
3. **候補抽出**: 住所が対象区内であることを確認（**目黒駅=品川区など、駅名に惑わされない**）
4. **包含基準判定**: 公式サイト/食べログで抹茶メニュー3品以上あるか確認。判定不明は「要確認」として含める。**この段階で判明した抹茶メニュー内容は `抹茶メニューリスト` 列に必ず格納**

### Step 2: GoogleマップURL取得（Playwright）
1. Playwright MCP で `https://www.google.com/maps/search/{店名}+{住所}` に navigate
2. 3秒待機（`browser_wait_for time:3`）
3. リダイレクト先URLが `/maps/place/...` 形式に解決していれば取得、そうでなければ検索クエリを調整してリトライ
4. **閉店店舗は検索してもplace URLに解決しない** → これをシグナルに閉店確認

### Step 3: ファイル追記（matcha_cafes.md のみ）
- 一覧表と詳細セクションの両方に追記
- **Phase A の最低限は id/店名/GoogleマップURL だが、この時点で判明した情報は全列格納**:
  - 住所、エリア、ブランド名、公式サイト、Instagram、食べログURL、抹茶メニューリスト等
- 判明していない項目のみ空欄
- **CSVは更新しない**（ユーザーが手動でやる）

### Step 4: 客観レビュー（各区完了ごと・必須）
1. `Agent` ツールで Explore または general-purpose サブエージェントを起動
2. プロンプトに「独立にリサーチして突合せよ。rubber stamp禁止。最低5件以上新規候補を挙げよ」と明記
3. 既存リストを見せた上で、漏れ・誤りを指摘させる
4. 指摘された新規候補を Step 2 で URL 取得して追記
5. 除外指摘があれば検証して反映

### Step 5: 中間報告 + ユーザー承認ゲート
- ユーザーに定型報告: 「追加N件 / 問題M件 / 除外M件 / サンプル5件」
- 承認を待ってから次の区へ

## 5. 作業時の注意事項（ハマりポイント集）

1. **駅名 ≠ 行政区**: 目黒駅は品川区上大崎、神楽坂駅は新宿区だが住所は「神楽坂」のみ表記のことがある。**必ず住所で区を判定**。
2. **WebSearch/WebFetch は動的ページに弱い**: Google 検索結果ページや Google Maps の place URL は JavaScript レンダリングで HTML に含まれない。place URL 取得には **必ず Playwright** を使う。
3. **サブエージェントが WebSearch 拒否する事例あり**: 「ツールがない」と主張して実行しない Explore エージェントが発生。プロンプトに「WebSearch / WebFetch を持っている。拒否・確認・言い訳禁止」と明記すること。それでも駄目なら親スレッドで直接実行。
4. **place URL に解決しない店**: 閉店、または店名が曖昧で検索ヒットが多い場合。検索クエリを詳細にする（店名+住所+区）。それでも解決しない場合は「未登録」として備考に理由を書く。
5. **重複排除はURLで**: 店名は表記揺れが多い。GoogleマップURLを正規化（クエリパラメータを除いた pathname 部分）して一意性判定する。
6. **チェーン店**: 1店舗1行。`店名`=支店込み正式表記、`ブランド名`列=集計用。
7. **推測・捏造禁止**: 不明値は必ず空欄。`情報ソース` 列にURL必須。
8. **「判明した情報は全部格納」**: Phase A 作業中でも、付随的に得た住所・メニュー・SNS等は必ず該当列に格納する。「後で埋める」は禁止。
9. **業態ミスマッチ注意**: 「懐石料理店」「居酒屋」等はカフェ業態ではないので対象外。八雲茶寮は客単価¥20-30kの懐石で除外。

## 6. 重要な運用ルール（CLAUDE.md からの抜粋）

- **2段階ワークフロー**: Phase A Step 1（店名+URL優先）→ Phase A Step 2（住所等追記）→ Phase B（属性リサーチ）
- **ただし判明情報は常時全列格納**（優先順位の話と空欄放置は別問題）
- **単一DB・単一テーブル**: `matcha_cafes.md` に一元化。1つの大きなテーブルで管理。詳細セクションをテーブル外に出すのは禁止
- **主出力は Markdown のみ（CSVなし）**: ユーザーが必要時に手動でMarkdownからコピペでSheets化
- **区切りごとに独立レビューサブエージェント必須**
- **家賃列は持たない**（信頼性ゼロ）
- **抹茶メニューリスト列(32)は必須に準ずる重要列**（包含基準の根拠）
- **確度列(33)で判定管理**: `確実 / 要確認 / 不明 / 除外候補`。迷ったら除外せず含める
- **営業状態列(34)で閉店店舗も含める**: `営業中 / 閉店 / 閉店予定`。閉店理由分析のため除外しない
- **サブエージェント大量並列は Playwright 不要な作業のみ**
